## Critical Issues (must fix)

- [ ] **Sister-CLI adoption is incomplete (AC-3, AC-4, AC-i3, AC-i6)**, `library/ledger/EXECUTION_LEDGER.md:22-24`

  The criteria require observed consumer behavior, not only a checklist. Live read-only grep across Doctor, Hive, Nectar, and Honeycomb still finds legacy top-level home paths and local home-resolution code; no fresh-install proof shows that one adopted CLI creates only `~/.apiary/`.

  ```text
  AC-4 ... fresh install of any one Apiary CLI creates only ~/.apiary/
  AC-i6 ... sisters delete their own home-resolution code and call the kit
  ```

- [ ] **JSON mode is not wired into an adopted CLI bootstrap (AC-g2)**, `src/json-output.ts:9-11`

  `setJsonMode()` correctly disables color, but the criterion also requires the CLI bootstrap to call it alongside color initialization. No consumer adoption proves that integration.

  ```ts
  export function setJsonMode(): void {
    disableColor();
  }
  ```

- [ ] **Published v0.2.0 release has no CycloneDX SBOM asset (AC-m3, AC-m4)**, `.github/workflows/release.yaml:79-80`

  The workflow defines SBOM generation and attachment, but `gh release view v0.2.0 --repo legioncodeinc/cli-kit --json assets` returned `"assets":[]`. Therefore the required release outcome and successful end-to-end CI execution are not proven.

  ```yaml
  - name: Generate CycloneDX SBOM
    run: npx --yes @cyclonedx/cyclonedx-npm@4.1.1 ...
  ```
